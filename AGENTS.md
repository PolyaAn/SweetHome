2. Технические спецификации, ТЗ, Jira

- Предпочтителен сжатый технический формат.
- Вводные формулировки не используются.
- Интерпретации не используются.
- Используются прямые информативные конструкции.
- Для ТЗ и текстов для Jira всегда используется защищённый формат:
  code block / plain text.
- Строка ---- сохраняется как текст и не преобразуется в горизонтальную линию.

----

3. Стиль ТЗ

- Формулировки максимально прямые и технические.
- Формулировка сразу фиксирует факт или требование.
- Не используются:
  - необходимо
  - следует
  - требуется
  - предполагается
  - должно быть реализовано
- Не описываются намерения, причины или ожидания.
- Фиксируется только поведение системы.
- Используются короткие однозначные предложения.
- Один пункт — одно правило.
- Исключается двусмысленность.
- Удаляются повторы и уточнения без влияния на логику.

----

4. Фокус требований

- Описывается:
  - что делает система
  - при каких условиях
  - какой результат

----

5. Недостающие детали

- Недостающие детали не выдумываются.
- Недостающие детали выносятся в раздел «Открытые вопросы».

----

6. Проверка финального ТЗ

- Перед финальным ТЗ текст проверяется на:
  - противоречия
  - неоднозначности
  - неполные сценарии

----

7. Git

- После внесения изменений изменения всегда заливаются в Git.
- Описание коммита пишется на русском.
- В описании коммита английский допускается только для наименований папок, файлов, разделов, методов и переменных.
